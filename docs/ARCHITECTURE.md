# Architecture

## 1. 设计原则

这个项目刻意把“研究判断”和“工程约束”分开：研究模块可以提出复杂假设，但回测引擎只接受结构化信号；因子可以改变置信度，但不能绕开风险否决；候选可以快速生成，但不能绕开验证集直接查看封存样本。

## 2. 数据边界

`demo.py` 生成固定的合成30分钟OHLCV。数据没有随机种子依赖，也不访问网络，因此任何机器运行都应得到相同结果。

真实研究接入行情时，需要额外处理交易日历、夜盘归属、主力合约切换、复权口径、涨跌停、手续费和滑点。本公开版不假装已经覆盖交易所级全部细节。

## 3. 常规因果框架

`causal_backtest.run_backtest` 分成两步：

1. 解析和验证所有信号，计算实际执行位置；
2. 按执行K线顺序推进持仓状态。

关键约束：

- `exec_bar_idx > signal_bar_idx`；
- 缺省执行位置为 `signal_bar_idx + 1`；
- 交易价格使用执行K线开盘价；
- 反向开仓先在同一可观察价格平掉旧方向，再建立新方向；
- 所有接受和拒绝均进入 `SignalCheck`。

`strategy_audit.py` 是静态保护层，不替代回测中的动态检查。它通过AST拦截负索引、`i + 1`型未来访问、危险导入以及 `eval/exec/open` 等调用。

## 4. 30/60分钟协同

`align_completed_30_60_bars` 每两根完整30分钟K线合成一根60分钟K线。如果最后只有一根30分钟K线，它会被明确记录为未完成，不进入慢周期判断。

协同逻辑保持简单：

- 慢周期方向为多，且快周期同向，允许产生多头候选；
- 慢周期方向为空，且快周期同向，允许产生空头候选；
- 其他情况等待。

这不是为了让两个周期重复确认同一个指标，而是把“环境方向”和“进入时机”拆开。

## 5. 小实体过滤

实体比例定义为：

```text
abs(close - open) / (high - low)
```

比例低于示例阈值的K线不参与有效趋势路径，但 `EffectiveBar.source_index` 始终指向原始K线。研究可以在压缩后的状态序列上判断，执行仍回到未压缩的真实时间轴，避免时间错位。

## 6. 因子协同

`combine_factors` 输出总分、每个贡献项、风险否决和文字解释。它体现三层关系：

1. 趋势决定方向；
2. 支撑压力、情绪、基本面和Local GEX调整置信度；
3. 风险条件决定是否允许执行。

Local GEX的示例处理是条件性的：负值增强当前趋势方向，正值削弱当前趋势方向；当趋势方向为零时，Local GEX贡献也为零。

## 7. 研究 Loop

`ResearchLoop` 保存父子候选关系。验证失败和封存失败分别记录阶段、分数和原因。封存样本具有一次性状态，防止研究者看到结果后反复修改同一候选。

它展示的是研究治理接口，而不是宣称一个简单评分可以证明策略真实有效。真实研究还需要滚动窗口、跨品种检验、成本敏感性、参数稳定性和市场状态归因。

## 8. 与实盘的边界

公开仓库停在“冻结候选版本”。研究系统不应直接操作账户：完整项目中应由独立、稳定的执行服务验证策略版本和哈希，管理订单状态、以柜台持仓为准进行对账，并在断线后优先恢复未完成的平仓任务。

这些内容只在这里说明架构边界，不公开真实执行代码和配置。
